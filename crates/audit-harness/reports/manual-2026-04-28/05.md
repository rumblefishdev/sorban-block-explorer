> E05 is the **exception case** in 0167's data-source boundary: the embedded `transactions[]` array is intentionally NOT served from the partitioned `transactions` table. Per ADR 0011 / ADR 0037 §11, it lives in a per-ledger S3 blob `s3://<bucket>/parsed_ledger_{sequence}.json`, with the DB's `sequence` as the bridge key.

## Step 1 — Run endpoint SQL

Single statement: header row from `ledgers` + `prev_sequence` / `next_sequence` via `LATERAL ... LIMIT 1` against `idx_ledgers_closed_at`.

3 sample sequences (head, mid, tail of 30k smoke range).

## Step 5 — Findings

### ✗ Mismatches

**Hash 3/3 vs Horizon — bug 0181, fixed on develop.** Same baked-in dataset issue as E04: this 30k smoke was indexed pre-0181, so all 30k stored hashes are wrong. Re-backfill against develop binary will restore canonical hashes.

### ⚠ S3 transactions[] blob — not yet populated

The SQL spec explicitly defers `transactions[]` to `parsed_ledger_{N}.json` on the explorer S3 bucket. **Grep across `crates/` returns no writer for this layout** (only an unrelated `build_parsed_ledgers` helper in `crates/api/src/contracts/handlers.rs`). This means:

- Frontend §6.6 "Transactions in ledger - paginated table" cannot be rendered today
- E05 endpoint cannot deliver the embedded list at all
- The ledger detail page is functionally incomplete until the indexer populates these blobs

**Owning task:** [0047 — Backend: Ledgers module](../../../../lore/1-tasks/active/0047_FEATURE_backend-ledgers-module.md) (active, karolkow). Worth confirming with @karolkow that the S3 blob writer is in scope. If it's split across an indexer-side task (write the blob during ingest) plus a backend-side task (E05 reads the blob), the split should be visible somewhere — currently it's not.

**No new task spawn from this audit** — flagging this back to 0047 owner instead.

### ⚠ prev/next at indexed-range boundaries

DB returns `prev_sequence = NULL` for sequence 62016000 (the head of OUR backfill, NOT the chain head) and `next_sequence = NULL` for 62046000 (tail of OUR backfill, NOT the chain tail). On real chain, both have neighbors.

Implication: the frontend "previous" / "next" navigation at backfill boundaries shows "no previous" / "no next" even when those ledgers exist on chain. This is an artifact of the indexing window, not a bug per se — but worth a frontend treatment so the user understands "no previous indexed" vs "no previous on chain". Could be a status badge or tooltip.

The SQL header note "Returning NULL when at the chain head/tail" is technically inaccurate — should read "at the indexed-range head/tail". Minor doc fix.

### ✓ Matches across DB + Horizon

`closed_at`, `protocol_version`, `transaction_count`, `base_fee` byte-for-byte for all 3 sample rows. Lateral prev/next correctly identifies adjacent indexed ledgers (consecutive sequence numbers, monotonic closed_at).
