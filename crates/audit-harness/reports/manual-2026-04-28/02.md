## Step 2 — Response shape vs frontend §6.3

Frontend §6.3 specifies columns:

| Frontend field | DB SQL output | Status |
| --- | --- | --- |
| hash | `encode(t.hash, 'hex')` | ✓ |
| ledger sequence | `t.ledger_sequence` | ✓ |
| source account | `src.account_id` | ✓ |
| operation type | `ops.operation_types[]` | ✓ |
| status (success/failed) | `t.successful` | ✓ |
| fee | `t.fee_charged` | ✓ |
| timestamp | `t.created_at` | ✓ |

DB also emits **extra fields** not in frontend §6.3:

- `application_order` — internal cursor / dedup, fine to keep
- `inner_tx_hash` — fee-bump info, **arguably belongs in frontend** (badge or tooltip), currently undocumented
- `has_soroban` — Soroban badge candidate, undocumented
- `contract_ids[]` — SQL header explicitly says "supports a future 'Touched contracts' column", documented intent
- `operation_count` — implicit in frontend usage but not in §6.3 column list

**Drift assessment:** non-blocking. Extra fields are forward-looking; frontend simply ignores them today. No data omitted that frontend needs. `has_soroban` and `inner_tx_hash` are good candidates for frontend §6.3 update (mention as optional badge / FB indicator).

## Step 5 — Findings

### ✓ Matches across DB + Horizon + (stellar.expert ledger)

Every queried field on every sampled row matches Horizon byte-for-byte. **Statement A on the no-filter path is correct on this dataset.**

### ⚠ Drift (non-blocking)

Frontend §6.3 spec doesn't mention `has_soroban`, `inner_tx_hash`, `contract_ids[]`. They're emitted by the SQL and (per SQL header) intended for future frontend use. **Recommendation:** update `frontend-overview.md` §6.3 to acknowledge these as optional/future columns so the contract is honest.

### ✗ Mismatches

None on E02 Statement A. The 30k smoke dataset has **no parse_error rows** in the random sample, no fee-bump source confusion, no fee_charged drift, no operation_count drift.

### Known caveats applied

- **`inner_tx_hash` for fee-bumps:** previously caught a false-positive harness bug (early Phase 2a comparing `fee_account.is_some()` instead of `inner_transaction.is_some()`). Manual audit re-confirmed: every row where Horizon says `inner_transaction` is set has our `inner_tx_hash` set, and every row where Horizon says it's null has ours null. ✓
- **Statement B (contract filter)** and **Statement C (op_type filter)** not exercised this turn — runbook step 3 says 2-3 random rows; we did 5 on the default path. Statement B is what task 0132 indexes (`idx_sia_contract_keyset` / `idx_sea_contract_keyset`) actually serve; worth a follow-up audit turn if a popular contract surfaces a slow plan.
