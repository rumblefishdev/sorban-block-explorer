## Step 1 — Run endpoint SQL

E03 is six statements: A (hash → ledger/created_at) → B (header) → C (operations) → D (participants) → E (invocations) → F (events). The API threads `(transaction_id, created_at)` from B into C-F.

- **Soroban:** hash `8fd56c14e646e4b5a7c0c34e439bad36b489544e6f2877cc1360308786e712f8` (ledger 62024073, op_count=1, has_soroban=true, no inner_tx)
- **Classic:** hash `a1a3a6e7900a9cb199411fba8c355a5297ad7e2b5317c4e9021a88ebf4b5a264` (ledger 62020687, op_count=2, has_soroban=false, no inner_tx)

## Step 5 — Findings

### ✓ Matches across DB + Horizon

Header (statement B) is byte-for-byte identical to Horizon for both sample tx. Statement A's hash → (ledger, created_at) routing returns the same ledger Horizon reports.

### ⚠ Convention differences (NOT bugs)

#### `operations_appearances` dedup (task 0163)

DB returns **1 row** for the classic tx with `operation_count=2` because both ops are identical-identity-tuple (same source, same null dest/asset/pool) and collapse per task 0163's appearance-index design. The fold count is in the `amount` column (= `appearance_count` semantically). Horizon shows 2 separate rows because it doesn't dedupe.

This is **documented behavior**. SQL spec at line 110-114 explicitly says `amount` is "a fold count of collapsed identity-tuple duplicates (task 0163), NOT a value". Frontend §6.4 Advanced mode should render appearance rows + per-row count, then optionally fetch full per-op detail from envelope_xdr (ADR 0029) for ops the user expands.

**Action:** verify frontend (when it lands) handles `appearance_count > 1` correctly. Not blocking E03 audit.

#### Operation source NULL when same as tx source

DB stores `oa.source_id = NULL` when an operation has no explicit `source_account` override (per Stellar XDR — operation source is `Optional<MuxedAccount>`). Horizon always resolves to the effective source (= tx source if not overridden).

This is a **storage convention**, not a bug. The API layer at the response edge can coalesce: `op.source_account = COALESCE(oa.source_account, tx.source_account)`. Worth confirming the API does this — search for usage in `crates/api/src/transactions/handlers.rs`.

### ✗ Mismatches

**None on either tx.**

### Caveats noted

- **memo / signatures / operation_tree** intentionally absent from DB per ADR 0029. Surfaced from envelope XDR at request time. Confirmed Horizon agrees on `memo_type=none` for the Soroban tx.
- **Statement F events for V4 meta:** Soroban tx had 2 event rows (XLM-SAC + called contract), which is non-zero — task 0173 V4 meta drop bug doesn't reproduce on this specific tx. The 91% no-events stat from Phase 1 was aggregate; individual tx may still be intact, depending on protocol version.
