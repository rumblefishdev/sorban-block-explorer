## Step 5 — Findings

### ✗ Mismatches

**Hash 3/3 mismatch on Horizon — same as bug 0176.** Already tracked. **No new task spawn.**

This audit confirms 0176 directly affects the user-facing E04 endpoint:

> Frontend §6.5 ledgers list shows `hash (truncated)`. Until 0176 lands, **every row in the ledger list page renders a wrong hash value**. Wrong enough that a user copying the truncated hash and looking it up on Horizon / stellar.expert will get 404.

→ Bumps 0176 priority. Worth flagging in the task body.

### ✓ Matches across DB + Horizon

`closed_at`, `protocol_version`, `transaction_count`, `base_fee` byte-for-byte on all 3 sample rows. The keyset cursor shape `(closed_at DESC, sequence DESC)` is consistent with the SQL spec; pagination should walk monotonically.

### ⚠ Drift

`base_fee` emitted by SQL but not in frontend §6.5 spec. Non-blocking — frontend ignores extra column. Worth either adding to §6.5 (since it's useful chain-wide info) or removing from SQL projection if truly unused.

### Caveats noted

Phase 2c archive XDR re-parse already independently confirmed bug 0176 at scale (50/50 ledgers mismatched). Manual audit on E04 just locks in that the bug is **frontend-visible** through this specific endpoint, not just an internal field.
