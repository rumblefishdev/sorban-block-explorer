## Step 5 — Findings

### ✗ Mismatches (now resolved on develop)

Hash 3/3 mismatch on Horizon — same root cause as task 0181 (already merged via PR #142). The 30k smoke dataset under audit was indexed by the pre-0181 binary, so the wrong hashes are baked in. A fresh backfill on the 0181'd develop will produce canonical hashes; re-running this audit on the new dataset is the regression-locking exercise.

Why the audit caught it three different ways: Phase 2a Horizon diff (50/50), Phase 2c archive XDR (50/50, independent extractor), and this manual E04 audit (3/3) all pointed at the same wrong-bytes-being-hashed extraction site (`crates/xdr-parser/src/ledger.rs`). Three independent paths agreeing was strong evidence the bug was real and not a Horizon / explorer staleness artifact.

### ✓ Matches across DB + Horizon

`closed_at`, `protocol_version`, `transaction_count`, `base_fee` byte-for-byte on all 3 sample rows. The keyset cursor shape `(closed_at DESC, sequence DESC)` is consistent with the SQL spec; pagination should walk monotonically.

### ⚠ Drift

`base_fee` emitted by SQL but not in frontend §6.5 spec. Non-blocking — frontend ignores extra column. Worth either adding to §6.5 (since it's useful chain-wide info) or removing from SQL projection if truly unused.
