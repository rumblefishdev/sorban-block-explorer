## Step 1 — Run endpoint SQL

## Step 5 — Findings

### ✗ Bug candidate: `sequence_number = 0` for active tx sources

Account `GDFAOY7JTTJYT5ZL73NJZYLKEWXTD6NLSFDHJY5AJ6EDX2A5BSPKIKLX` was the source of **340 transactions** in the 30k window (62016000-62046000) but `accounts.sequence_number = 0` in our DB. Stellar protocol bumps `sequence_number` on every successful tx sourced by that account, so:

- Either the parser is not extracting `AccountEntry` state changes for these accounts (despite the entry being modified on every tx-source bump), or
- The persist layer is coalescing `-1` → `0` even when an extracted state should have written the real value.

The `accounts` upsert at `crates/indexer/src/handler/persist/write.rs:31` uses sentinel `-1` for "no state override" and `COALESCE(NULLIF(sq, -1), 0)` SQL on new rows. If `account_states` from `xdr-parser::extract_account_states` is empty for tx-source accounts, the upsert path takes the no-override branch and persists `0`.

Phase 1 invariant `accounts.I3` (`first_seen ≤ last_seen`) reports 0 violations on this dataset, but no Phase 1 invariant currently checks that "an account that's a tx source ⇒ sequence_number > 0" — worth adding as a sibling invariant once the bug is confirmed.

### ⚠ Drift (acceptable, not bugs)

- **Snapshot vs current state for balance amounts** — DB carries balances at `last_updated_ledger` within the 30k window; Horizon shows current state from full chain. Documented in `manual-endpoint-audit.md` §"Snapshot vs current state".
- **Pre-window trustlines** — sample 2 has 13 balances on Horizon vs 12 in DB. The missing trustline (EURC) was last modified at ledger 60702946, BEFORE our window. Trustlines that don't change within the window are invisible to the 5-day snapshot. Closes on full re-backfill from genesis.

### ✓ Matches

- Response shape covers every frontend §6.7 field.
- Sample 2 balance set: 11/12 credit balances match Horizon (issuer strkey + asset code).
- Statement B's `LEFT JOIN accounts iss` correctly yields `NULL` for native rows (sample 2 row 1).
- Sample 3 (lobstr.co) shows `home_domain` is correctly surfaced.
- Sample 3 sequence_number is non-zero and matches the order of magnitude expected for an active account.

## Bug spawns

**1 candidate.** To spawn after user-side confirmation on real wallets they care about:

- **`sequence_number = 0` for tx-source accounts** — needs reproduction on user-picked StrKey + binary log of `extract_account_states` output for one of the 340 tx sourced by sample 1. If the extractor never emits an `AccountEntry` state for these txs, root cause is in `crates/xdr-parser/src/state.rs`. If it emits but persist writes 0 anyway, root cause is in `crates/indexer/src/handler/persist/write.rs`.

## Cross-link to automated harness

- Phase 1 `accounts.I1` (StrKey shape): 0 violations on 30k ✓ (proves no C-prefix / oversize keys)
- Phase 1 `accounts.I2` (account_id UNIQUE): 0 violations ✓
- Phase 1 `accounts.I3` (first_seen ≤ last_seen): 0 violations ✓
- Phase 2a `horizon-diff --table accounts --sample 100` on 30k: 0/100 mismatched, 5 unreachable on Horizon (Soroban-only accts, expected). Note the harness compares `home_domain` and `signers` shape but not `sequence_number` (Horizon shows current vs our snapshot — would always drift). The sequence-number-zero bug therefore slips past the automated harness.
- **Suggested follow-up invariant:** "every account that appears as `transactions.source_id` ≥ 1 time in any partition ⇒ `accounts.sequence_number > 0`". Add to `audit-harness/sql/11_accounts.sql` if the user confirms the bug.
