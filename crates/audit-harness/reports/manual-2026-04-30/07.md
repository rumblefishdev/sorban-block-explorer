# E07 — `GET /accounts/:account_id/transactions`

**Date:** 2026-04-30
**Auditor:** stkrolikiewicz (following [`manual-endpoint-audit.md`](../../../../lore/3-wiki/manual-endpoint-audit.md))
**Dataset:** 30k smoke (mainnet ledgers 62016000–62046000), develop binary
post entire Apr 30 fix stack.
**SQL spec:** [`07_get_accounts_transactions.sql`](../../../../docs/architecture/database-schema/endpoint-queries/07_get_accounts_transactions.sql)
**Frontend spec:** [`frontend-overview.md` §6.7](../../../../docs/architecture/frontend/frontend-overview.md) (reuses §6.3 transactions list shape)

## Step 1 — Run endpoint SQL

Single statement with leading `acc` CTE that resolves `account_strkey →
accounts.id`. Walks `transaction_participants` PK in `(account_id,
created_at DESC, transaction_id DESC)` order and joins to `transactions`
via the composite `(id, created_at)` FK. Lateral subquery aggregates
`operation_types` from `operations_appearances`. Keyset cursor passes
`(created_at, tx_id)` with row-value comparison.

All 3 picks return rows; runs cleanly with `LIMIT 5` page-1 and a follow-up
page-2 cursor walk.

## Step 5 — Findings

### ✓ Matches

- Response shape covers every §6.7 / §6.3 column the frontend renders.
- Keyset pagination walks `(created_at, tx_id)` strictly monotonically
  across page boundary; same-`created_at` rows correctly sub-ordered by
  `tx_id`; page 2 cursor reads picked up where page 1 left off, no
  duplicates and no gaps.
- `transaction_participants` includes the source account (per ADR 0020 /
  task 0163 ingestion contract), so this endpoint covers tx-source rows
  via the same join — no UNION with `transactions.source_id` required;
  spec note is accurate.
- `has_soroban` flag is exclusively `true` on `INVOKE_HOST_FUNCTION`
  rows in the sample 2 mix — classic txs correctly tag `false`.
- `operation_types` lateral correctly returns `DISTINCT … ORDER BY name`
  arrays (`{PAYMENT}`, `{PATH_PAYMENT_STRICT_SEND}`, `{INVOKE_HOST_FUNCTION}`).

### ⚠ Drift / Notes (not bugs)

- The `cursor_tx_id` column (BIGINT surrogate from `transactions.id`) is
  internal — not rendered by the frontend, but encoded into the keyset
  cursor token. Out of scope for the end-user view; documented in spec
  header.
- Sample 1 has zero successful transactions on its 5-row page (all
  `successful = false`). This is genuinely the data — every recent tx
  involving GDFAOY in 2026-04-09 was a failed `PATH_PAYMENT_STRICT_*`
  attempt against this address. Likely an exchange / sweep account
  hammered by no-op probes or rate-limited bot traffic. Not a bug.
- The earlier-spawned task 0185 (E06 finding) does NOT affect this
  endpoint's projection — `accounts.sequence_number` is not surfaced
  here; only `transactions.*` and `transaction_participants` rows are
  read. E07 stays correct independently of how 0185 resolves.

### ✗ Mismatches

None on the 2 detail-cross-checked tx hashes. Sample 3 left as an
exercise for the user-side spot check.

## Hand-off

Per runbook §"Step 5 — Stop, report findings, hand back": **stop**. User to:

1. Pick a real wallet StrKey (own or known-active mainnet) and re-run:

    ```bash
    curl -s "https://horizon.stellar.org/accounts/G.../transactions?limit=5&order=desc" \
      | jq '.[] | {hash, source_account, successful, operation_count, fee_charged}'

    PGPASSWORD=postgres psql -h localhost -p 5432 -U postgres -d soroban_block_explorer <<'SQL'
    WITH acc AS (SELECT id FROM accounts WHERE account_id = 'G...')
    SELECT LEFT(encode(t.hash,'hex'),16) AS hash, src.account_id AS source,
      t.successful, t.operation_count, t.fee_charged, t.created_at
    FROM acc
    JOIN transaction_participants tp ON tp.account_id = acc.id
    JOIN transactions t ON t.id = tp.transaction_id AND t.created_at = tp.created_at
    JOIN accounts src ON src.id = t.source_id
    ORDER BY tp.created_at DESC, tp.transaction_id DESC LIMIT 5;
    SQL
    ```

2. If the user picks an account whose recent tx involve Soroban,
   verify `has_soroban=true` and `operation_types` includes
   `INVOKE_HOST_FUNCTION` for those rows.

3. After confirmation, proceed to **E08** (`GET /assets`).
