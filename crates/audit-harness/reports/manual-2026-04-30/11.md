## Step 1 — Run endpoint SQL

Two-statement endpoint. **A** = contract header (deployer, WASM hash,
type, SAC flag, full metadata JSONB). **B** = stats window
(`recent_invocations`, `recent_unique_callers` over `NOW() - $2`).
Runs cleanly on both samples.

### Source B: Statement B (stats window) on retrospective dataset

```sql
SELECT COUNT(*) AS recent_invocations, COUNT(DISTINCT sia.caller_id) AS recent_unique_callers,
       '7 days'::interval AS stats_window
FROM soroban_invocations_appearances sia
WHERE sia.contract_id = $1 AND sia.created_at >= NOW() - '7 days'::interval;
-- recent_invocations = 0 | recent_unique_callers = 0
```

The window predicate `created_at >= NOW() - '7 days'::interval` evaluates
against `NOW()` = today (`2026-05-01`); our 30k window data is from
`2026-04-07`–`2026-04-09`, which is more than 7 days old, so the stats
return 0 regardless of how heavily the contract was used inside the window.

**This is the spec design** — the header explicitly says "we deliberately
bound the window so this never becomes a full-history scan", and the FE
labels the count with the chosen window label. **For production live
data the stats are accurate and useful.** For retrospective audit
against a frozen 30k smoke, the numbers are always zero — that's a
property of the design, not a bug.

## Step 5 — Findings

### ✓ Matches

- Statement A returns the full header set when the contract was deployed
  inside the window (sample 2). Deployer, deployed_at_ledger, is_sac,
  contract_type all populated, decoded names correct.
- BLTC SAC's `deployed_at_ledger = 62042818` matches the
  ledger-close timestamp on stellar.expert's `created` unix field
  byte-for-second.
- `is_sac=true` correctly surfaced as a direct stored boolean (not
  derived from contract_type), matching the spec's "do not derive"
  rule.
- `wasm_hash = NULL` on SAC sample 2 is semantically correct (SAC
  contracts have no WASM upload by design).

### ⚠ Drift / known gaps (not bugs)

- **Pre-window-deployed contracts have NULL header fields** (sample 1):
  4.1M invocations against `CDL74RF5…IGWA` in our window, but `wasm_hash`,
  `deployer_id`, `deployed_at_ledger`, `contract_type` are all NULL
  because the deploy event happened in November 2024 — well before our
  backfill window begins. The contract row was created implicitly on
  first invocation. UX implication: production T0 (full chain backfill
  from genesis) closes this. Mid-deployment audit datasets (5-day smoke
  etc.) will always show "ghost" rows for any pre-window-deployed
  contract that's actively invoked in-window.
- **`stats_window` returns 0 on retrospective data** (Statement B):
  the `NOW() - $2` predicate always misses our 5-day window once the
  data ages past 7 days. Spec design — accurate for live API,
  no-op for stale audit. The audit harness's automated path can't
  meaningfully test these stats numbers; only live integration tests can.
- **`contract_type` NULL on most rows** (13 664 of 13 756 contracts):
  task 0118 (WASM-spec classifier) only sets type on contracts whose
  WASM matched a known interface signature. Pre-window-deployed
  contracts have NULL `wasm_hash` so the classifier can't fire even
  if the interface is recognizable. Same root cause as the prior
  bullet; same fix (full backfill).

### ✗ Mismatches

None against protocol-truth on the in-window sample. Pre-window NULLs
are documented snapshot artifacts.

## Bug spawns

**None new.** The "ghost contract" issue is a known limitation of the
audit dataset, not a defect in this endpoint. Worth tracking as a
follow-up backlog item if not already covered: "ensure
`/contracts/:id` for pre-window-deployed contracts gracefully renders
'deployment data not indexed in this dataset' rather than blank
fields" — UX-side concern, not a SQL or correctness one. Decide
post-T0 once the backfill range is full.
