## Step 1 ‚Äî Run endpoint SQL

CTE-driven query: `matched_ops` filters `operations_appearances` by
`pool_id` (uses partial `idx_ops_app_pool` directly), DISTINCT-ON
deduplicates same-tx multi-op LP touches, outer JOIN to `transactions`
+ source `accounts`. Lateral aggregates op_types per tx so FE can
categorise trade-vs-LP-mgmt. Runs cleanly.

## Step 2 ‚Äî Response shape vs frontend ¬ß6.14

Same column shape as E07 / E10 (transactions list). FE uses
`operation_types` array to colour rows by category:

- **LP-mgmt:** `LIQUIDITY_POOL_DEPOSIT`, `LIQUIDITY_POOL_WITHDRAW`
- **Trade:** `PATH_PAYMENT_*`, `MANAGE_*_OFFER`, `CREATE_PASSIVE_*`

Categorisation is FE policy, not SQL.

## Step 5 ‚Äî Findings

### ‚úì Matches

- Partial index `idx_ops_app_pool` correctly used (WHERE keeps
  `pool_id = $1` predicate, partial predicate `WHERE pool_id IS NOT
  NULL` satisfied).
- DISTINCT ON dedup handles multi-op tx (rows 3-5 each have 3 ops but
  appear once).
- `operation_types` lateral correctly returns sorted DISTINCT array;
  FE has the data needed for trade-vs-LP-mgmt categorisation.
- Horizon match on row 1 (5/5 fields).
- Mix of failed (row 3) + successful tx in window ‚Äî endpoint returns
  both, FE filters as needed.
