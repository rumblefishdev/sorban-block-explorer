# E13 — `GET /contracts/:contract_id/invocations`

**Date:** 2026-04-30
**Auditor:** stkrolikiewicz (following [`manual-endpoint-audit.md`](../../../../lore/3-wiki/manual-endpoint-audit.md))
**Dataset:** 30k smoke (mainnet ledgers 62016000–62046000), develop binary
post entire Apr 30 fix stack (incl. lore-0183 invocation coverage from
fn_call/fn_return diagnostic events).
**SQL spec:** [`13_get_contracts_invocations.sql`](../../../../docs/architecture/database-schema/endpoke-queries/13_get_contracts_invocations.sql)
**Frontend spec:** [`frontend-overview.md` §6.10](../../../../docs/architecture/frontend/frontend-overview.md) (Invocations tab)

## Step 1 — Run endpoint SQL

Single statement: `soroban_contracts → soroban_invocations_appearances`
on the leading equality, joining `transactions` via composite FK,
LEFT-joining both `accounts` (for G/M caller) and `soroban_contracts`
(for C-prefix caller). Keyset on `(ledger_sequence DESC, transaction_id
DESC, created_at DESC)` served by `idx_sia_contract_ledger`. Runs cleanly.

## Step 4 — Cross-check

### Source A: Caller-split distribution (post-0183 invocation coverage)

```sql
SELECT
  COUNT(*) FILTER (WHERE caller_id IS NOT NULL)          AS account_caller,
  COUNT(*) FILTER (WHERE caller_contract_id IS NOT NULL) AS contract_caller,
  COUNT(*) FILTER (WHERE caller_id IS NULL AND caller_contract_id IS NULL) AS no_caller,
  COUNT(*) AS total
FROM soroban_invocations_appearances sia
JOIN soroban_contracts sc ON sc.id = sia.contract_id
WHERE sc.contract_id = 'CDL74RF5…IGWA'
  AND sia.created_at >= '2026-04-07' AND sia.created_at < '2026-04-10';

  account_caller | contract_caller | no_caller |  total
 ----------------+-----------------+-----------+---------
         3611605 |          508566 |         0 | 4120171
```

- **88 % G-account callers** (top-level invocations from EOA tx sources).
- **12 % contract callers** (sub-calls from DeFi router → this contract,
  surfaced via diagnostic-event execution tree).
- **0 % no-caller** — every row has exactly one caller side populated,
  matching the `ck_sia_caller_xor` invariant.

The 12 % contract-caller share would have been **invisible pre-lore-0183**
(when only the auth tree was indexed). Post-fix this endpoint correctly
surfaces the full call graph including auth-less internal invocations —
exactly what task 0183 was scoped to deliver.

### Source B: stellar.expert tx detail (sample row 1)

```bash
curl -s "https://api.stellar.expert/explorer/public/tx/92fc6463…68a8"
```

Returned `hash: 92fc6463…68a8`, `ledger: 62046000` — both match our
DB. The stellar.expert public tx endpoint returns sparse fields for
this Soroban tx (no per-call invocation breakdown via the public API);
deeper signature-level cross-check would require the indexer's
read-time XDR re-parse, out of scope for this row-shape audit.

## Step 5 — Findings

### ✓ Matches

- Post-lore-0183 caller split is working: `caller_id` (G/M) and
  `caller_contract_id` (C) populated per the `ck_sia_caller_xor`
  schema constraint; LEFT JOINs on both surface either side cleanly
  with NULL on the other.
- Caller-split distribution (88 / 12 / 0) matches the expected shape
  for a high-traffic DeFi contract: most invocations are top-level EOA
  calls, a meaningful tail are sub-calls from router contracts.
- 0 % no-caller rows confirm the diag-tree extraction always assigns a
  caller (either auth-root tx source for top-level, or parent contract
  for sub-call). No drift from the schema's at-most-one rule.
- Keyset pagination strictly monotonic across page boundary
  (`(62046000, 10119107) < (62046000, page-1-min)`), no duplicates,
  no gaps unrelated to the contract's actual tx footprint.
- Sample tx hash `92fc6463…68a8` exists on stellar.expert at ledger
  62046000 — the canonical tx existence is confirmed.

### ⚠ Drift / notes (not bugs)

- **Function name / args / return value not projected** — by spec
  design (ADR 0029 / 0034 read-time XDR). The frontend's invocations
  tab shows the row metadata (caller, ledger, status, timestamp, fold
  count); function-level detail comes from a separate read-time
  archive XDR fetch on click. No mismatch.
- **Sparse stellar.expert API fields on Soroban tx** — `successful`
  and `created` came back NULL through the public tx endpoint. Their
  UI clearly shows them (Soroban Explorer page renders both), so the
  data is there on stellar.expert's side, just not exposed by this
  particular API path. For audit purposes the hash+ledger match is
  enough.

### ✗ Mismatches

None.

## Bug spawns

**None new.** The post-0183 invocation coverage is functioning as
designed; the 12 % contract-caller share that would have been missing
pre-fix is now correctly surfaced.
