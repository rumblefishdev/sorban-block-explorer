## Step 1 — Run endpoint SQL

Single statement: keyset on `(created_at_ledger DESC, pool_id DESC)`,
LEFT JOIN issuer accounts (per leg), and a LATERAL on
`liquidity_pool_snapshots` to fetch the latest snapshot within the
freshness window. Five filters (asset_a_code, asset_a_issuer,
asset_b_code, asset_b_issuer, min_tvl) plus pagination. Runs cleanly.

## Step 5 — Findings

### ✓ Matches

- Shape covers every §6.13 column.
- `pool_id` byte-for-byte matches stellar.expert.
- `fee_bps = 30` matches; the projected `fee_percent = 0.30` correctly
  divides bps → percent on the SQL side.
- Asset legs (codes + types) match.
- Canonical order verified at scale: `native_b = 0` across all 16k
  pools (native always asset_a when present, per task 0179's I3 + PR
  #151's protocol-hash confirmation).
- LATERAL snapshot lookup works when the window covers the data
  (`>= '2026-04-07'`); returns NULL on the spec's default `1 day`
  window because our 30k smoke is 22+ days old.

### ⚠ Drift / known gaps (not bugs)

- **`created_at_ledger` semantic on TTL-restored pools.** Pool
  `b67d79fb…` was created on chain **2024-09-26** but our DB shows
  `created_at_ledger = 62045992` (2026-04-09 21:52). Root cause: per
  `crates/xdr-parser/src/state.rs:460`, the parser flags
  `is_creation = true` on both `created` AND `restored` change types
  — Stellar's TTL archive cycle restores pool entries on use, so
  every pool that hadn't been touched in a few months gets a
  "creation"-stamped event when a swap revives it from archive. On
  this 30k smoke, 16,007 pools effectively means "pools observed in
  any state-change event during the 5-day window", not "pools created
  for the first time on chain in those 5 days". Misleading column
  name; UX render of "Created at" should clarify or pivot to a
  "first observed in indexed range" label until full backfill closes
  the gap.
- **`tvl` NULL on all rows** — TVL computation pending (task 0135
  related). FE renders "—" gracefully.
- **`min_tvl` filter cannot be exercised** on this dataset because
  `tvl IS NULL` everywhere; passes the predicate-shape check but
  matches no rows. Closes once TVL aggregation lands.
- **Snapshot freshness window default 1 day** is by design useful for
  live data but useless for retrospective audit (same pattern as E11's
  recent-window stats). Spec design.

### ✗ Mismatches

None against protocol-truth. `pool_id`, fee_bps, asset legs all match
stellar.expert. The `created_at_ledger` semantic is a labelling
concern, not a data correctness concern.
