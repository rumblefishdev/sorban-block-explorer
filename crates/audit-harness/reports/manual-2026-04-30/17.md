## Step 1 — Run endpoint SQL

Single statement walking `nft_ownership` PK on `(nft_id =, created_at
DESC, ledger_sequence DESC, event_order DESC)`. `LAG(owner)` window
function reconstructs from-owner. Joins `transactions` for tx hash
display. Runs cleanly — but returns 0 rows on every input.

## Step 4 — Cross-check

### Source A: code path — `nft_ownership` is empty by design

`crates/indexer/src/handler/persist/write.rs:1477` carries the comment:

```rust
// 12b. nft_ownership (empty until parser catches up)
```

And `staging.rs:982-989` builds the `nft_ownership_rows` Vec from the
`nft_events` parser input. The parser's `nft_events` extraction has not
yet landed — confirmed by:

```sql
SELECT COUNT(*) AS total_events FROM nft_ownership;
-- 0
```

Yet `nfts.current_owner_id` IS populated for **654 090 of 1 175 493
NFTs** (55.6 %). Two paths feed the NFT subsystem:

| Surface | Source | Status today |
| --- | --- | --- |
| `nfts.current_owner_id` (snapshot) | parser's contract-storage walk in `extract_nfts` | ✓ Working — every ledger's storage snapshot updates the watermark |
| `nft_ownership` (event history) | parser's `nft_events` (mint/transfer/burn) emission | ✗ Not yet emitted — pending parser work |

The endpoint shape is correct but the upstream data is empty.

## Step 5 — Findings

### ✓ Matches

- SQL shape, indexes, keyset cursor design all correct per spec.
- `LAG()` window function correctly walks the same DESC ordering as
  the result set; from-owner synthesis logic is sound.
- LEFT JOIN on accounts handles burn rows (owner_id NULL) cleanly.
- `nfts.current_owner_id` snapshot path works (E15 / E16).

### ⚠ Drift / known gap (not a bug)

- **`nft_ownership` table is empty** despite `nfts.current_owner_id`
  being populated for 654k NFTs. Documented in
  `write.rs:1477` ("empty until parser catches up"). The parser's
  `nft_events` extraction has not landed; `nfts` snapshot is currently
  the only NFT data the indexer emits.
- **UX implication on the FE:** the Transfers tab on the NFT detail
  page will be empty for every NFT until the events extractor lands.
  Frontend should either hide the tab entirely or show a clear "Transfer
  history not yet indexed" placeholder rather than a misleading "no
  transfers" empty state.
- The cross-page-from-owner stitch logic the spec describes (passing
  `prev_owner_strkey` between pages) is correctly designed but
  cannot be exercised on this dataset.

### ✗ Mismatches

None against the documented contract. Empty rows match the documented
"empty until parser catches up" state.
