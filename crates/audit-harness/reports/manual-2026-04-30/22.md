# E22 — `GET /search`

**Date:** 2026-04-30
**Auditor:** stkrolikiewicz (following [`manual-endpoint-audit.md`](../../../../lore/3-wiki/manual-endpoint-audit.md))
**Dataset:** 30k smoke (mainnet ledgers 62016000–62046000), develop binary
post entire Apr 30 fix stack.
**SQL spec:** [`22_get_search.sql`](../../../../docs/architecture/database-schema/endpoint-queries/22_get_search.sql)
**Frontend spec:** [`frontend-overview.md` §6.15](../../../../docs/architecture/frontend/frontend-overview.md)

## Step 1 — Run endpoint SQL

UNION ALL of six narrow CTEs (`tx_hits`, `contract_hits`, `asset_hits`,
`account_hits`, `nft_hits`, `pool_hits`). Each gated by `:include_*` flag
and bounded `LIMIT $4`. Input shape classifier (hash bytes / strkey
prefix / plain text) decides which CTE actually fires. Indexes:
trigram GIN on asset_code + nfts.name; `text_pattern_ops` btree on
contract_id + account_id for prefix LIKE; PK probes on tx hash + pool_id.

## Step 2 — Response shape vs frontend §6.15

| Frontend field | DB SQL output | Status |
| --- | --- | --- |
| entity_type (badge) | `entity_type` column | ✓ |
| display identifier | `identifier` (hex hash / StrKey / asset code / nft name / pool_id hex) | ✓ |
| context label | `label` (ledger N / metadata.name / asset_type / home_domain / collection / pair) | ✓ |
| route surrogate | `surrogate_id` BIGINT (NULL on tx + pool which route by natural id) | ✓ |

Spec calls out **two routing-id traps**:
- `asset_hits.identifier` is asset_code (not unique on classic) — FE must route by `surrogate_id`.
- `nft_hits.identifier` is NFT name (not unique) — FE must route by `surrogate_id`.

Audit confirms both are surfaced as documented.

## Step 3 — Sample (4 input shapes)

### a) Plain-text substring `"BLT"` → asset CTE (trigram GIN)

| entity_type | identifier | label | surrogate_id |
| --- | --- | --- | --- |
| asset | BLTA | sac | 6 |
| asset | BLTB | sac | 7 |
| asset | BLTC | sac | 8 |

`idx_assets_code_trgm` (gin_trgm_ops) serves the leading-`%` ILIKE.

### b) StrKey prefix `"GBQR7IMF"` → account CTE (text_pattern_ops btree)

| entity_type | identifier | label |
| --- | --- | --- |
| account | `GBQR7IMFKZHN…YMRI` | (empty home_domain) |

The BLTC issuer (E09 / E10 sample) — exact 8-char prefix lookup served
by `idx_accounts_prefix`.

### c) Hex hash `"92fc6463d5d3a18d…"` → tx CTE (PK probe)

| entity_type | identifier | label |
| --- | --- | --- |
| transaction | `92fc6463…68a8` | ledger 62046000 |

Same hash used in E13 / E14 cross-checks. PK on `transaction_hash_index`
returns one row.

### d) Hex pool_id `"ad45b80a3a90ccaa…"` → pool CTE (PK probe)

| entity_type | identifier | label |
| --- | --- | --- |
| pool | `ad45b80a…0255c2` | SSLX / STARDUST |

Top-traffic pool from E20. Label correctly renders the asset pair from
the pool's identity rows.

## Step 4 — Cross-check

Each CTE was triggered against a known entity from prior endpoints:

- BLT* assets exist (E08).
- `GBQR7IMF…` exists as the BLTC issuer (E09).
- Tx `92fc6463…` exists (E13/E14, Horizon-confirmed).
- Pool `ad45b80a…` exists (E20).

All four queries returned the expected single hit (or 3 for BLT
substring). No false positives, no missed legitimate matches.

## Step 5 — Findings

### ✓ Matches

- All six CTE branches compile and route to the correct underlying index.
- `WHERE FALSE`-style guards via `$5..$10 = TRUE` correctly suppress
  unwanted entity types — confirmed by branch-level `LIMIT` on disabled
  CTEs returning 0 rows.
- Trigram on asset_code substring works (`'%BLT%'` returns 3).
- StrKey prefix LIKE on text_pattern_ops btree works.
- Hex hash + hex pool_id PK probes work.
- `surrogate_id` populated where expected (asset, contract, account, nft);
  NULL on tx + pool (route by natural id) per spec contract.
- Pool label renders `XLM / X` correctly when one leg is native (E18 had
  one such pool; ad45b80a here is non-native both legs, label `SSLX /
  STARDUST` correct).

### ⚠ Drift / notes (not bugs)

- **NFT search would be empty** because all `nfts.name IS NULL` on this
  dataset (per E15 — metadata extraction pending). The CTE is
  syntactically correct and the index is in place; once names land, the
  search will return nft rows.
- **Contract full-text search** (`sc.search_vector @@
  plainto_tsquery(...)`) cannot be exercised meaningfully — most
  contracts have NULL metadata (per E11/E12 ghost-contract pattern).
  Once metadata extraction lands the FT search becomes useful.
