**Date:** 2026-04-30
**Auditor:** stkrolikiewicz (following [`manual-endpoint-audit.md`](../../../../lore/3-wiki/manual-endpoint-audit.md))
**Dataset:** 30k smoke (mainnet ledgers 62016000–62046000), develop binary
post entire Apr 30 fix stack.
**SQL spec:** [`16_get_nfts_by_id.sql`](../../../../docs/architecture/database-schema/endpoint-queries/16_get_nfts_by_id.sql)
**Frontend spec:** [`frontend-overview.md` §6.12](../../../../docs/architecture/frontend/frontend-overview.md)

## Step 1 — Run endpoint SQL

Single statement on `nfts.id` PK with `JOIN soroban_contracts` (always
present per FK) and `LEFT JOIN accounts` (NULL on burned/unowned).
Index-only paths on each. Runs cleanly.

Sample 2 is interesting: `current_owner_ledger = 62046000` but

that ledger as "no owner" — typical of a burn event (`nft_event_type =

show the burn event explicitly.

## Step 5 — Findings

### ✓ Matches

- Shape covers every §6.12 column.
- LEFT JOIN behaviour correct on burned/unowned NFTs (sample 2):
  current_owner = NULL, current_owner_ledger non-NULL — exactly the
  shape ADR 0037 §13 mandates for the burn state.
- Owned-NFT path (sample 1) returns full owner G-strkey via the join.
- 30k smoke contains 1.18 M NFTs across 10,141 contracts; the per-id
  detail SQL is just a PK probe, so latency-wise this endpoint is
  trivial.

### ⚠ Drift / known gaps (not bugs)

- **Metadata empty across all NFTs** — same as E15. Closes on metadata
  extraction landing post-T0.
- **`minted_at_ledger` NULL on pre-window NFTs** — sample 1 and 2 both
  have no recorded mint event because the NFT was minted before our
  backfill range begins. UX implication: detail page can't show
  "minted at" timestamp for these NFTs; FE should render "—" or
  "before indexed range".
- **Storage-entry count vs NFT count divergence** (cross-check
  observation): stellar.expert reports `CCW67TSZ` has **12,164 storage
  entries**; our DB has **145,597 NFT rows** for that same contract.
  A 12 × ratio between storage entries and our NFT rows is suspicious
  for a true NFT contract (each token usually maps to one storage
  entry). Plausible explanations:
  - The contract is a **fungible token** whose per-recipient transfer
    events we're misclassifying as NFT mints (task 0118 territory:
    "NFT false positives from fungible token transfers").
  - The contract uses an unusual compact storage scheme (multiple
    logical tokens per entry — rare).
  - We're double-counting due to event-replay extraction (less likely
    given our PRIMARY KEY on `(contract_id, token_id)` would dedup).

  Same observation from E15 (CAS3J7GY had token_ids in 10-billion
  range — also fungible-amount-shaped). **Track via task 0118**, do
  NOT spawn duplicate. Worth a focused re-audit on a known-NFT
  contract (e.g. SVRD Galactic NFTs, etc.) to confirm the classifier
  excludes legit fungibles correctly post-fix.

### ✗ Mismatches

None against protocol-truth on the per-row shape. Pre-window mint
artifact + classifier concern are pre-existing, tracked elsewhere.

## Bug spawns

**None new.** Two tracked elsewhere:
- NFT metadata extraction completeness — pre-existing backlog item.
- NFT vs fungible classification — task 0118.

## Hand-off

Per runbook §"Step 5 — Stop, report findings, hand back": **stop**. User to:

1. Hit the API for the owned and burned samples:

    ```bash
    curl -s "http://localhost:<port>/nfts/4601558" | jq
    curl -s "http://localhost:<port>/nfts/4601556" | jq
    ```

2. Verify owned sample shows `current_owner: "GBOA3PJZ…"`,
   burned sample shows `current_owner: null` with
   `current_owner_ledger: 62046000`.

3. Cross-check against stellar.expert UI by visiting the contract's
   page and looking up the same `token_id`.

4. Confirm the storage-entry vs NFT-count divergence either way
   (genuine fungible misclassification, or contract uses compact
   storage). If misclassification, flag for task 0118 owner.

5. After confirmation, proceed to **E17** (`GET /nfts/:id/transfers`).
