# E08 — `GET /assets`

**Date:** 2026-04-30
**Auditor:** stkrolikiewicz (following [`manual-endpoint-audit.md`](../../../../lore/3-wiki/manual-endpoint-audit.md))
**Dataset:** 30k smoke (mainnet ledgers 62016000–62046000), develop binary
post entire Apr 30 fix stack.
**SQL spec:** [`08_get_assets_list.sql`](../../../../docs/architecture/database-schema/endpoint-queries/08_get_assets_list.sql)
**Frontend spec:** [`frontend-overview.md` §6.8](../../../../docs/architecture/frontend/frontend-overview.md)

## Step 1 — Run endpoint SQL

Single-statement keyset on `id DESC` with three optional filters
(`asset_type`, `asset_code` ILIKE, `cursor_id`). Joins `accounts` for
issuer StrKey (NULL on native + soroban-native) and `soroban_contracts`
for contract StrKey (NULL on native + classic_credit) per the
asset-identity check. Runs cleanly against the 30k DB.

## Step 2 — Response shape vs frontend §6.8

§6.8 wants: asset code, issuer or contract ID, type (native / classic
credit / SAC / Soroban), total supply, holder count + filters by type
and asset code, cursor pagination.

| Frontend field | DB SQL output | Status |
| --- | --- | --- |
| asset code | `a.asset_code` | ✓ |
| issuer (G-strkey, classic_credit only) | `iss.account_id AS issuer` (LEFT JOIN) | ✓ |
| contract ID (C-strkey, SAC + soroban) | `sc.contract_id AS contract_id` (LEFT JOIN) | ✓ |
| type (native / classic_credit / SAC / soroban) | `token_asset_type_name(a.asset_type)` | ✓ |
| total supply | `a.total_supply` | ⚠ NULL on this dataset (see findings) |
| holder count | `a.holder_count` | ⚠ NULL on this dataset (task 0135 — token holder count tracking) |
| icon URL | `a.icon_url` | ✓ projection (no spec render rule but UI uses it) |
| asset name | `a.name` | ✓ projection |
| asset_type filter | WHERE `a.asset_type = $3` (idx_assets_type) | ✓ |
| asset_code substring filter | WHERE `a.asset_code ILIKE '%'||$4||'%'` (gin_trgm_ops) | ✓ |
| cursor pagination | `a.id` keyset (idx_assets_pkey) | ✓ |

Spec note: `holder_count` "may be NULL or stale: ongoing tracking is
blocked behind task 0135". Aligns with our observation of 0/8 rows
populated.

## Step 4 — Cross-check

### Source A: stellar.expert / Horizon (sanity)

Horizon `/assets?asset_code=AQUA&order=desc&limit=1` returned a real
AQUA asset (issuer `GDZGPS33…`). The same asset appears in our
`account_balances_current` for sample 2's wallet (issuer `GBNZILSTV…` —
note: different issuer; multiple AQUAs exist on chain), but **does NOT
appear in `assets` table** because the trustline existed before our
30k window and no creation event fired within `[62016000, 62046000]`.

Sanity scan on the discrepancy:

```sql
SELECT COUNT(DISTINCT (asset_code, issuer_id))
FROM account_balances_current WHERE asset_type != 0;
-- 12 110
```

vs our `assets` table having `0` rows of `asset_type = 1` (classic
credit). The 7 SACs and 1 native survive because they're either
contract-deployment-driven (SAC) or schema-seeded (native).

### Source B: SDF Anchored Assets list

Skipped — would only confirm the hypothesis ("assets table is sparse
because the 5-day window excludes pre-existing trust creations"),
which the local SQL above already proves.

## Step 5 — Findings

### ✓ Matches

- Shape covers every §6.8 column.
- Issuer / contract LEFT JOINs correctly yield NULL on native (row 1)
  and on SACs return both (row 2-8) per the asset-identity contract
  (issuer = the original Stellar G-key that created the SAC; contract
  = the SAC's deployed `C…` address).
- `token_asset_type_name(asset_type)` decoder returns expected names
  (`native`, `sac`).
- All three filters (type, code-substring, cursor) execute under the
  declared indexes (`idx_assets_type`, `idx_assets_code_trgm`,
  `assets_pkey`) without sequential scans.
- Trigram substring search with leading `%` returns the expected
  3 rows for `%BLT%`, validating that `gin_trgm_ops` is doing its job
  (a btree on `asset_code` would have returned 0 rows for that pattern).

### ⚠ Drift / known gap (not a bug)

- **Sparse classic-credit registry on a 5-day window.** `assets`
  table has only 8 rows on the 30k smoke; on-chain there are
  ≥ 12 110 distinct `(code, issuer)` pairs visible to our wallets'
  trustlines. The asset-extraction path appears to register classic
  credits only on FIRST trustline creation / asset-issuance event,
  which for ~all popular assets (AQUA, USDC, yXLM, …) happened
  before our window. Closes naturally on a full re-backfill from
  genesis. UX implication: a 5-day-only deployment of `/assets`
  would show 8 rows; production T0 (full mainnet history) will show
  the expected tens of thousands. Documented limitation, not a bug.
- **`total_supply` and `holder_count` NULL across the board** —
  expected per spec note ("may be NULL or stale: ongoing tracking is
  blocked behind task 0135"). Frontend should render these gracefully
  (em-dash / "—") until 0135 lands. Not an E08-side concern.

### ✗ Mismatches

None against the protocol-truth surface. The 5-day-window sparsity is
documented and on-track for closure with full backfill.

## Bug spawns

**None new.** The thin assets-registry coverage is an expected
characteristic of the audit dataset, not a defect in this endpoint or
its SQL. Task 0135 (token-holder-count tracking) and the ADR
asset-extraction-from-trustlines work track the underlying coverage
gaps separately; both are pre-existing.
