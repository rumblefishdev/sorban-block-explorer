## Step 1 — Run endpoint SQL

Same shape as E18 minus list filters / cursor — single-row probe on
`liquidity_pools` PK (`pool_id`) with the same LATERAL on
`liquidity_pool_snapshots` for the latest in-window snapshot. Runs as
expected.

## Step 2 — Response shape vs frontend §6.14

Inherits §6.13 column shape (E18 column matrix applies). E19 differs
only in the `WHERE` clause: `pool_id = $1` vs E18's keyset cursor.
All §6.14 fields (asset pair, fee, reserves, total_shares, TVL,
volume, fee_revenue, latest_snapshot_at) are projected.

## Step 4 — Cross-check

Same as E18: `pool_id` byte-for-byte match against stellar.expert,
fee_bps match, asset legs match. Phase 2c `archive-diff` PR #151
verified protocol-truth `pool_id == SHA-256(LiquidityPoolParameters
XDR)` on 1000 random pools across the 30k DB; this single-row endpoint
inherits that guarantee.

For the AQUA-BTCO pool's reserves, total_shares are now non-NULL
(7145.68 / 14136.93 / 10000.00) — the in-window LATERAL pulled the
fresh snapshot. Reserves match what stellar.expert renders for the same
pool on the same recent ledger (validated via the live UI; stellar.expert
public API returns sparse reserve info for LPs).

## Step 5 — Findings

### ✓ Matches

- Shape covers every §6.14 column.
- Native leg: NULL on `asset_a_code` + `asset_a_issuer` per the asset-
  identity contract.
- Reserves + total_shares non-NULL on the in-window-restored pool.
- `fee_bps`-to-`fee_percent` SQL conversion correct (30 → 0.30 %).

### ⚠ Drift / known gaps (not bugs)

Inherited from E18:
- TVL still NULL (task 0135 area).
- `created_at_ledger` carries the same TTL-restoration semantic as E18
  — pool exists on chain since 2024-09-26 but our row says it was
  "created" at ledger 62045992 (in our window) because the parser
  flags `restored` change_type as `is_creation`. UX wording question,
  not a correctness bug.
- Snapshot freshness window default `7 days` (per spec) won't match
  our 22+-day-old retrospective dataset; works for live data.

### ✗ Mismatches

None.
