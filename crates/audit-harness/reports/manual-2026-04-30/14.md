# E14 — `GET /contracts/:contract_id/events`

**Date:** 2026-04-30
**Auditor:** stkrolikiewicz (following [`manual-endpoint-audit.md`](../../../../lore/3-wiki/manual-endpoint-audit.md))
**Dataset:** 30k smoke (mainnet ledgers 62016000–62046000), develop binary
post entire Apr 30 fix stack.
**SQL spec:** [`14_get_contracts_events.sql`](../../../../docs/architecture/database-schema/endpoint-queries/14_get_contracts_events.sql)
**Frontend spec:** [`frontend-overview.md` §6.10`](../../../../docs/architecture/frontend/frontend-overview.md) (Events tab)

## Step 1 — Run endpoint SQL

Single statement: `soroban_contracts → soroban_events_appearances` on
leading equality, joining `transactions` via composite FK. Keyset on
`(ledger_sequence DESC, transaction_id DESC, created_at DESC)` served
by `idx_sea_contract_ledger`. Per ADR 0033, the appearance index is
DB-only — full event topics/data/event_type come from the archive XDR
at read-time (ADR 0029).

## Step 2 — Response shape vs frontend §6.10

| Frontend field | DB SQL output | Status |
| --- | --- | --- |
| transaction hash | `encode(t.hash, 'hex')` | ✓ |
| ledger sequence | `sea.ledger_sequence` | ✓ |
| successful flag | `t.successful` | ✓ |
| amount (folded events count for trio) | `sea.amount` | ✓ ADR 0033 |
| created_at timestamp | `sea.created_at` | ✓ |
| transaction_id (cursor key) | `sea.transaction_id` | ✓ |
| event_type / topics / data | NOT in DB — Archive XDR overlay (ADR 0029, 0033) | ✓ documented |

Frontend's "Events tab" lists per-event metadata. The DB row gives the
identifying triple `(ledger_sequence, transaction_id, created_at)` plus
contract — enough for the API to fan out to the archive XDR slice and
filter the parsed `events[]` array down to this contract.

## Step 4 — Cross-check

### Source A: Cross-link with E13 invocations

Sample row 1 hash `92fc6463d5d3a18d…` appears in BOTH:

- **E13** (invocations on `CDL74RF5…` router): caller_contract = `CBGSBKYM…`,
  amount = 2 (the router invoked CDL74RF5 twice in this tx)
- **E14** (events on `CAS3J7GY…` token): amount = 2 (the token emitted
  2 events in this tx)

This consistency is exactly what the appearance-index design predicts
— the same tx that triggered router → CDL74RF5 sub-calls also hit a
classic token (`CAS3J7GY…`) twice, each emitting 2 events. The
contracts are different but appear in the same tx via the same call
graph.

### Source B: stellar.expert tx detail

Sample tx `92fc6463…68a8` already cross-checked under E13: hash present,
ledger 62046000 confirmed. Per-event topic/data fidelity is the read-
time archive XDR concern (`crates/api/...` overlay path), out of scope
for this DB-side audit.

## Step 5 — Findings

### ✓ Matches

- Appearance-index shape per ADR 0033: one row per
  `(contract_id, transaction_id, ledger_sequence, created_at)` trio with
  `amount` folding the per-tx event count for this contract.
- Keyset cursor served by `idx_sea_contract_ledger`; descending walk
  monotonic; same pattern verified under E13 for invocations applies
  identically here.
- Same-tx cross-link to E13 invocations is internally consistent: the
  router-invocation path and the event-emission path point to the same
  underlying tx (`92fc6463…`), with both `amount`s reflecting per-tx
  fold counts.
- 10M+ event appearance rows for the top emitter, distributed across
  the 30k ledger window, all tagged `t.successful = true` on page 1
  (events fire on successful execution; failed-tx events would be a
  rare edge case worth a separate spot-check post-T0).

### ⚠ Drift / notes (not bugs)

- **`event_type` / `topics` / `data` not in DB.** Per spec — they come
  from archive XDR at read-time. Out of scope for this row-shape
  audit. Worth a follow-up integration test that exercises the
  archive overlay end-to-end on this exact tx_hash to validate that
  the API layer correctly filters `parsed_events[]` to this
  contract.
- **Pre-window-deployed contract (`CAS3J7GY…`)** — same ghost pattern
  as E11 / E12: the contract has 10M+ event rows in our window, but
  `soroban_contracts.deployed_at_ledger` is NULL because the deploy
  event was outside the backfill range. Snapshot artifact, closes on
  full re-backfill.
