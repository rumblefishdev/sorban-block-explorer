# E12 — `GET /contracts/:contract_id/interface`

**Date:** 2026-04-30
**Auditor:** stkrolikiewicz (following [`manual-endpoint-audit.md`](../../../../lore/3-wiki/manual-endpoint-audit.md))
**Dataset:** 30k smoke (mainnet ledgers 62016000–62046000), develop binary
post entire Apr 30 fix stack.
**SQL spec:** [`12_get_contracts_interface.sql`](../../../../docs/architecture/database-schema/endpoint-queries/12_get_contracts_interface.sql)
**Frontend spec:** [`frontend-overview.md` §6.10](../../../../docs/architecture/frontend/frontend-overview.md) (Contract Interface panel)

## Step 1 — Run endpoint SQL

Single statement: `soroban_contracts → wasm_interface_metadata` LEFT JOIN
on `wasm_hash`. Both joins are PK / UNIQUE-key probes, so the planner
hits index-only paths on each. SAC contracts (no WASM) yield NULL
metadata via the LEFT JOIN; pre-window-deployed contracts whose WASM
upload event happened outside our backfill range also yield NULL.

## Step 2 — Response shape vs frontend §6.10

§6.10 wants "list of public functions with parameter names and types,
allowing users to understand the contract's API without reading source
code".

| Frontend field | DB SQL output | Status |
| --- | --- | --- |
| contract_id | `sc.contract_id` | ✓ |
| wasm_hash (debug / link) | `encode(sc.wasm_hash, 'hex')` | ✓ |
| interface JSONB (`functions[]`) | `wim.metadata` | ✓ shape, ⚠ JSONB shape not formalized in `docs/architecture/**` (spec note) |

The spec calls out a **schema-doc gap**: `wasm_interface_metadata.metadata`
JSONB is set by the indexer at ingest time (parsed from the WASM
custom section) but its layout (`{ "functions": [{ "name", "inputs",
"outputs", … }] }`) isn't documented in ADR 0037. The endpoint
returns the JSONB verbatim and the frontend is expected to be defensive
about missing keys. Out of scope for this audit; a doc follow-up worth
spawning (low priority — not blocking shipping).

## Step 4 — Cross-check

### Source A: stellar.expert API

#### Sample 1 — `CDUS7NSF…O6NK` (in-window deploy)

| field | DB | stellar.expert | match |
| --- | --- | --- | --- |
| contract id | `CDUS7NSF…O6NK` | `CDUS7NSF…O6NK` | ✓ |
| wasm (first 32 hex) | `414c07c64ee4a814a81387e4e5269632` | `414c07c64ee4a814a81387e4e5269632` | ✓ |
| creator | (not projected by E12 — covered by E11) | `GBKDPWX3…` | (out of scope here) |
| created (unix) | (not projected — out of scope) | `1775599537` = `2026-04-07T22:05:37Z` | (in our window — confirms in-window deploy) |
| function count parsed | 3 (`exec_op`, `withdraw`, `prepare_arb`) | 0 (this stellar.expert endpoint does not return parsed signatures; likely served from a different surface) | (not directly comparable) |

The WASM hash matches byte-for-byte with stellar.expert's recorded
hash, validating our indexer's WASM extraction path. The function
signatures themselves cannot be cross-checked through the stellar.expert
public API used here (it returns `functions: []` — UI fetches from a
different endpoint we don't have), so signature-level fidelity stays
at the spec-and-shape level.

#### Sample 2 — `CB2J5FHC…2OGK` (SAC, no WASM)

| field | DB | expected | match |
| --- | --- | --- | --- |
| wasm_hash | NULL | NULL | ✓ |
| metadata | NULL (LEFT JOIN no row) | NULL | ✓ |

Per CAP-0046-A, SAC contracts use host-built native code with no WASM
upload. `NULL` on both columns is the semantically correct response.
Frontend should render "No interface declared (Stellar Asset Contract)"
or fall back to a synthesized stub (per spec note).

#### Sample 3 — `CDL74RF5…IGWA` (pre-window deploy)

| field | DB | expected | match |
| --- | --- | --- | --- |
| wasm_hash | NULL | (real WASM `db2c14290d4964e3…` per stellar.expert) | ⚠ snapshot artifact |
| metadata | NULL | (would be real interface if WASM upload were in window) | ⚠ same |

E11's "ghost contract" pattern repeats here: any contract deployed
before our 30k window has a `soroban_contracts` row created implicitly
on first invocation appearance, with NULL wasm/wasm_uploaded_at. E12
faithfully renders that NULL — frontend has nothing to display in the
interface panel for these contracts. Closes on full re-backfill.

### Source B: stellarchain.io

Skipped — stellar.expert's WASM hash match already confirmed the
canonical extraction. Function-signature parser fidelity is best
audited via the indexer's own test fixtures, not external comparison.

## Step 5 — Findings

### ✓ Matches

- WASM hash on sample 1 byte-for-byte match against stellar.expert.
- Sample 1 deploy timestamp 2026-04-07 22:05 UTC falls inside our
  backfill window — confirms in-window deploy path captures both
  WASM upload event AND interface parsing.
- LEFT JOIN behavior correct on both NULL paths (SAC + pre-window
  ghost): no spurious "metadata" rows; FE gets clean NULL signal.
- JSONB shape on sample 1 includes `name`, `inputs[].name`,
  `inputs[].type_name`, `outputs[]` — exactly the shape §6.10's
  "function name + parameter names + types" rendering needs.
- 85 contracts have WASM hash; 66 have wasm_interface_metadata rows
  (ratio 85→66 means ~22 % of WASM contracts have non-parsed interface,
  worth investigating but not in scope here).

### ⚠ Drift / known gaps (not bugs)

- **Pre-window-deployed contracts return NULL interface** (sample 3):
  same root cause as E11 — deploy event outside our backfill range.
  Closes on full re-backfill. UX implication: ghost-contract rows on
  the contract page show no interface panel content even when the
  contract is recognized on-chain.
- **22 % of WASM contracts have no interface metadata row** (`85 -
  66 = 19` contracts): these are contracts whose WASM was uploaded
  in-window but our parser didn't successfully parse the custom
  section (corrupt section, version we don't know, etc.). Worth
  spawning as a separate investigation if the gap persists at full-
  backfill scale; on the 30k smoke 19 / 13756 = 0.14 % of contracts,
  not blocking. Could also be racing with WASM-upload-via-newer-
  protocol; hard to tell without deeper sampling.
- **JSONB shape not in `docs/architecture/**`** — spec note flagged
  this; worth a doc follow-up (similar shape to lore-0184). Low
  priority.

### ✗ Mismatches

None against the protocol-truth surface. WASM hash and deployment
timestamp both match stellar.expert byte-for-byte / second-for-second.

## Bug spawns

**None new.** The 22 % wasm-without-metadata gap is interesting enough
to track if it persists post-T0 (full-backfill); at 0.14 % of all
contracts it's not blocking. Would surface naturally as a Phase 1
invariant follow-up: "every contract with `wasm_hash IS NOT NULL`
has a matching `wasm_interface_metadata` row" — currently absent
because the schema permits the gap (LEFT JOIN handles it gracefully).
