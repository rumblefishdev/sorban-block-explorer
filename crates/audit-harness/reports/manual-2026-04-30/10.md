## Step 1 — Run endpoint SQL

Two-variant query: **A** (classic identity, hits `idx_ops_app_asset` via
`(asset_code, asset_issuer_id)`) and **B** (contract identity, hits
`idx_ops_app_contract` via `contract_id`). The API picks one or merges
both based on asset type. SAC has both identities; classic_credit only
A; soroban_native only B; native has neither (FE falls back to global
recent or empty list).

Both variants exercised on the BLTC SAC (`id=8`) and return rows with
the expected partial-index predicates intact.

## Step 5 — Findings

### ✓ Matches

- Both variants execute under their declared partial indexes
  (`idx_ops_app_asset`, `idx_ops_app_contract`); WHERE clauses retain
  the `asset_code IS NOT NULL` / `contract_id IS NOT NULL` predicates so
  the planner stays on the partial.
- `DISTINCT ON (created_at, transaction_id)` correctly collapses
  multi-op tx (e.g. an op_count=99 row would otherwise appear 99×) into
  one row per tx.
- Inner `LIMIT $2 * 4` then outer `LIMIT $2` gives the join enough
  rows to survive the dedup without scanning the full partition.
- Variant A rows are cleanly disjoint from Variant B rows for the BLTC
  sample — classic identity (ChangeTrust ops by issuer) at ledger
  ~62042815, contract identity (Soroban invocations by holders) at
  ~62043569. Validates the two-path design: each variant captures a
  distinct, non-overlapping slice of the asset's lifecycle.
- Horizon match on Variant A row 1: hash, source, successful, op_count,
  ledger — all five fields byte-for-byte identical.

### ⚠ Drift / notes (not bugs)

- API-side merge logic (the spec mentions "the API SHOULD merge results
  from both variants…or pick one based on the UX — typically classic
  for SAC") is out of scope for the SQL audit. SQL ships both variants
  cleanly; `crates/api/src/assets/handlers.rs` decides the merge/pick.
  Worth a follow-up code-review pass that the merge produces a
  monotonic combined keyset cursor when both variants are surfaced.
- Native (asset_type=0) intentionally has no SQL path here — confirmed
  by the spec note. Frontend handles via fallback.
