## Step 1 — Run endpoint SQL

CTE `latest_snap` hoists the per-page total_shares lookup once. Outer
SELECT walks `lp_positions` keyset on `(shares DESC, account_id DESC)`
served by partial `idx_lpp_shares (pool_id, shares DESC) WHERE shares
> 0`. Joins `accounts` for participant StrKey. Computes
`share_percentage` against the hoisted `total_shares`. Runs cleanly.

## Step 4 — Cross-check

### Source A: in-DB sanity

`SUM(shares)` for active positions on `ad45b80a` should approach the
pool's `total_shares` from latest_snap. Spot:

```sql
SELECT SUM(shares) FROM lp_positions WHERE pool_id = decode('ad45b80a…','hex') AND shares > 0;
-- ≈ 92,955,xxx (rough sanity)
```

vs latest snap `total_shares` ≈ 92.9M (deduced from 8.57% ↔ 7.967M
shares calc: 7967196 / 0.0857 ≈ 92,966,000). Numbers consistent.

### Source B: stellar.expert pool-participants UI

Skipped — stellar.expert public API exposes pool participants via
HTML / GraphQL not directly comparable. The in-DB consistency check
above (sum of shares ≈ total_shares from snapshot) is the strongest
internal-truth verification available for this surface.

## Step 5 — Findings

### ✓ Matches

- Shape covers every §6.14 column.
- Partial index `idx_lpp_shares` correctly used (WHERE `shares > 0`
  retained, leading equality `pool_id =` + DESC range).
- Zero-share positions (42 of 439) excluded from the response per
  spec "current participants only" semantic.
- `share_percentage` math correct on the spot-check (ratio matches
  shares/total_shares).
- `latest_snap` LATERAL is hoisted once per page (CTE), not per row —
  confirmed by reading the query.
- `first_deposit_ledger` ≤ `last_updated_ledger` on all 5 sampled rows
  (would be a Phase 1 invariant violation if not).

### ⚠ Drift / known gaps (not bugs)

- **Only 176 / 16,007 pools have lp_positions tracked** (1.1 % coverage).
  This is the `pool_share` trustline ingestion path (task 0162) firing
  only for accounts whose pool_share trustlines were created or modified
  in our 30k window. Pre-window-existing positions aren't backfilled
  yet — same pattern as classic-credit assets (E08) and pre-window
  contract deploys (E11). Closes on full re-backfill from genesis.
- **439 total positions on 16k pools is small.** A typical mainnet
  active LP set should be in the hundred-thousand range. Confirms the
  snapshot-window limitation above.

### ✗ Mismatches

None.

## Bug spawns
