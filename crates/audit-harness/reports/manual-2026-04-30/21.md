## Step 1 — Run endpoint SQL

Time-bucketed aggregation over `liquidity_pool_snapshots` filtered by
`pool_id` + `created_at` range. `date_trunc` with bucket keyword
mapped from user-facing `'1h' | '1d' | '1w'` via inline CASE. Aggregation
policy: TVL = last value in bucket (ARRAY_AGG ORDER BY DESC + index 1);
volume / fee_revenue = SUM. Runs cleanly.

## Step 2 — Response shape vs frontend §6.14

| Frontend field | DB SQL output | Status |
| --- | --- | --- |
| time bucket | `date_trunc(...)` | ✓ |
| TVL at close of bucket | `(ARRAY_AGG(tvl ORDER BY created_at DESC))[1]` | ⚠ NULL |

| fee_revenue in bucket | `SUM(lps.fee_revenue)` | ⚠ NULL |

## Step 5 — Findings

### ✓ Matches

- Shape covers every §6.14 chart column.
- Bucketing fires correctly for both `1d` and `1h` intervals.
- `samples_in_bucket` non-zero on every bucket — aggregation is
  receiving rows.
- ARRAY_AGG-ORDER-BY pattern for "TVL at close of bucket" parses
  correctly (and would yield real values once tvl is populated).
- ORDER BY bucket ASC matches FE's chart-rendering expectation
  (oldest → newest left-to-right on chart).

### ⚠ Drift / known gaps (not bugs)

- **TVL / volume / fee_revenue NULL on every row.** Same root cause as
  E18/E19: the LP TVL+volume aggregation pipeline hasn't landed (task
  0135 area). Frontend chart will render blank lines until data is
  computed.
- **Defensive ELSE branch in CASE** — spec says the API allowlist
  validates `$2` upstream; the CASE has no ELSE, so a bad value would
  cause `date_trunc(NULL, ...)` and probably an error. Left as-is per
  spec ("fail loudly is the correct posture"); no change here.

### ✗ Mismatches

None.
