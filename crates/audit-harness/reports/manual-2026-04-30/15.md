**Date:** 2026-04-30
**Auditor:** stkrolikiewicz (following [`manual-endpoint-audit.md`](../../../../lore/3-wiki/manual-endpoint-audit.md))
**Dataset:** 30k smoke (mainnet ledgers 62016000–62046000), develop binary
post entire Apr 30 fix stack.
**SQL spec:** [`15_get_nfts_list.sql`](../../../../docs/architecture/database-schema/endpoint-queries/15_get_nfts_list.sql)

## Step 1 — Run endpoint SQL

Single statement keyset on `id DESC` with three optional filters
(`collection_name` exact match via `idx_nfts_collection`, `contract_id`
resolved via leading CTE, `name` substring via `idx_nfts_name_trgm`).
Joins `soroban_contracts` for the C-strkey and `accounts` for the
current-owner G-strkey. Runs cleanly.

## Step 5 — Findings

### ✓ Matches

- Shape covers every §6.11 column.
- Surrogate `id DESC` keyset works; `contract_id` filter via CTE
  resolves correctly (145,597 rows for CCW67TSZ matches direct join
  count).
- `LEFT JOIN accounts own` correctly yields NULL for unowned/burned
  NFTs; `with_owner / total = 654,090 / 1,175,493 = 55.6 %` is a
  plausible on-chain ratio (some NFT mints, transfers in flight, or
  burns).
- Top NFT contracts by token count look reasonable for mainnet
  (collections in the 40k-220k range — consistent with high-volume
  NFT projects).

### ⚠ Drift / known gaps (not bugs)

- **Metadata fields all NULL** (`name`, `collection_name`, `media_url`,
  `metadata` JSONB). Two possible reasons:
  1. NFT metadata extraction (parse SEP-39-style JSON from contract
     storage / events) hasn't landed yet — pre-existing backlog.
  2. The pre-window-deployed contracts' metadata-write events
     happened outside the backfill window.

  Either way, the filters that use these columns (`collection_name=`,
  `name ILIKE`) cannot be exercised on this dataset; they pass the
  index check but won't match any rows. UX implication: the FE list
  view will show contract+token_id+owner only on this dataset; once
  metadata extraction lands, the same SQL surfaces full nft cards.

- **NFT classifier output empty** (`contract_type NULL` on all 5 top
  NFT contracts). Same pattern as E11/E12 — classifier needs WASM
  interface metadata, which is missing for pre-window-deployed
  contracts. Closes on full backfill.

- **521,403 NFTs without owner** (44.4 %). Could be: real burn
  state on chain, real "issued but unminted" placeholders, or a
  partial extraction of transfer events that left current_owner_id
  NULL when the mint event was missed. Worth a deeper sub-audit
  cross-checked against stellar.expert NFT history (out of scope
  here — would be a focused 0118-style follow-up if the ratio
  holds suspicious post-T0). Phase 1 invariants don't currently
  cover an "owner_ledger NOT NULL ⇒ owner_id NOT NULL" rule; would
  be a candidate to add.

- **NFT vs fungible classification.** Top contract CAS3J7GY has
  222k tokens with numeric `token_id`s ranging into 10-billion-
  scale (`10541238299`) — token_id values that high suggest the
  contract may be a fungible token whose per-balance entries are
  being misclassified as NFTs (task 0118 territory). This 30k
  smoke pre-dates the 0118 fix landing, so some false positives
  may persist; worth a separate spot-check against a known fungible
  token (e.g. AQUA SAC) to confirm the filter is correctly excluding
  it now. **Not surfaced as a new bug here** — task 0118 already
  tracks this class, and the 30k pre-fix data is the wrong dataset
  to confirm regression.

### ✗ Mismatches

None against protocol-truth — pre-window-deployed contracts'
sparse-metadata pattern is documented snapshot behavior, not a
defect in this endpoint.

## Bug spawns

**None new.** Two pre-existing concerns to track via post-T0
re-audit on a full-backfill dataset:

1. NFT metadata extraction completeness (name / collection_name /
   media_url all NULL).
2. NFT classifier on contracts with token_id values that look
   fungible-amount-shaped (CAS3J7GY's 10-billion-scale token_id).

## Cross-link to automated harness

- Phase 1 `nfts.*` (UNIQUE on (contract_id, token_id), FK validity for
  contract + current_owner): 0 violations on 30k ✓
- Phase 1 `nft_ownership.*` (FK validity, partition routing, monotonic
  event_order): 0 violations on 30k ✓
- Phase 2a — Horizon doesn't index NFTs. stellar.expert is the
  closest external source.
- **Suggested Phase 1 invariant** post-T0 (when metadata extraction
  lands): "every nft row has either both `(name, media_url)` non-NULL
  OR is from a contract where the metadata-extractor explicitly
  flagged 'no per-token metadata schema'". Today this invariant would
  be 100 % violation, gating it to "live" datasets is the right move.
