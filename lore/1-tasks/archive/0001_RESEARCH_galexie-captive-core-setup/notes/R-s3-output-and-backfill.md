# S3 Output Format, Checkpoint Restart, and Backfill Model

Sources researched: Galexie admin docs, source code (stellar/stellar-galexie), stellar/go SDK datastore schema, stellar.org blog posts.

---

## 1. S3 Key Format

### Path Structure

The key is generated by `GetObjectKeyFromSequenceNumber` in `go/support/datastore/schema.go`. The full path pattern is:

```
{bucket}/{destination_bucket_path}/{partition_dir}/{file}
```

Where:

**Partition directory** (only when `files_per_partition > 1`):

```
{hex(uint32_max - partition_start_ledger)}--{partition_start}-{partition_end}/
```

**File name** (single ledger per file):

```
{hex(uint32_max - file_start_ledger)}--{file_start_ledger}.xdr.zst
```

**File name** (multiple ledgers per file):

```
{hex(uint32_max - file_start_ledger)}--{file_start_ledger}-{file_end_ledger}.xdr.zst
```

The hex prefix is 8 characters, zero-padded, uppercase. `uint32_max = 4294967295 = 0xFFFFFFFF`.

### Concrete Example (from monitoring logs)

With `ledgers_per_file = 1`, `files_per_partition = 64000`, ledger 200:

- `partition_size = 1 * 64000 = 64000`
- `partition_start = (200 / 64000) * 64000 = 0`
- `partition_end = 0 + 64000 - 1 = 63999`
- `hex(4294967295 - 0) = FFFFFFFF`
- `file_start = 200`, `file_end = 200` (single ledger)
- `hex(4294967295 - 200) = FFFFFF37`

Key: `FFFFFFFF--0-63999/FFFFFF37--200.xdr.zst`

For comparison, the official monitoring docs (source: `stellar-docs-galexie-monitoring.md`) show this log output from a **different configuration**:

```
Uploading: FFFFFF37--200-299/FFFFFF37--200.xdr.zstd
```

The exact `ledgers_per_file` and `files_per_partition` values that produced this log are not documented. The `FFFFFF37` hex prefix corresponds to ledger 200 (`uint32_max - 200 = 0xFFFFFF37`), and `200-299` is a partition or file range of 100 ledgers. Note: `schema.go` only generates a partition directory when `FilesPerPartition > 1`, so this log was produced with `files_per_partition > 1`.

> **Extension note**: The file extension is definitively **`.xdr.zst`** — confirmed by `DefaultCompressor.Name()` returning `"zst"` in `support/compressxdr/compressor.go` (source: `github-galexie-extension-verification.md`). The `.xdr.zstd` shown in the official monitoring docs log example is outdated documentation — it reflects a legacy extension. The `schema.go` `FileExtension` field comment says "Optional – for backward (zstd) compatibility only", confirming `.zstd` is the old format. Galexie v23.0+ rejects datastores containing files with non-matching extensions.

### Purpose of the Hex Prefix

The hex prefix is `uint32_max - ledger_number`. This puts higher (more recent) ledgers at lower hex values, causing S3/GCS to list them in descending chronological order when lexicographically sorted. Newest data sorts first.

## 1b. XDR Content Format

Each exported file contains a single **`LedgerCloseMetaBatch`** XDR object. This is a container type that holds one or more `LedgerCloseMeta` entries:

- With `ledgers_per_file = 1`: the batch contains exactly **one** `LedgerCloseMeta` (the common case).
- With `ledgers_per_file = 64`: the batch contains **64** `LedgerCloseMeta` entries.

The downstream Ledger Processor Lambda must parse the file as a `LedgerCloseMetaBatch` (not a raw `LedgerCloseMeta`), then iterate over the entries inside the batch. This is the same format regardless of whether the data comes from live ingestion or historical backfill.

## 2. Compression Format

**zstd (Zstandard)** is confirmed as the sole compression format. Hardcoded constant in `internal/config.go`:

```go
const compressionType = "zstd"
```

The file extension is `.xdr.zst` (the compressor's canonical name). Galexie v23.0+ requires an empty datastore; older files using a different extension (e.g. `.xdr.gz`) are rejected with an error:

```
detected older incompatible ledger files in the data store (extension "...").
Galexie v23.0+ requires starting with an empty datastore
```

## 3. Single Ledger vs. Batched

Configurable via `ledgers_per_file` in `[datastore_config.schema]`. Options:

| Setting                 | Behavior                                                            |
| ----------------------- | ------------------------------------------------------------------- |
| `ledgers_per_file = 1`  | One ledger per file (public data lake default: pubnet S3 uses 1)    |
| `ledgers_per_file = 64` | Batch of 64 (blog post notes this is ~2x faster for bulk downloads) |

The file name encodes the range: `{start}.xdr.zst` for single, `{start}-{end}.xdr.zst` for batched.

`files_per_partition` controls the directory grouping (e.g., 64000 = one dir per 64,000 files). The default values in `config.example.toml` are `ledgers_per_file = 1, files_per_partition = 64000` (source: `github-galexie-config-example-toml.md`).

## 4. Checkpoint-Aware Restart Mechanism

### How `append` mode resumes

The `append` subcommand has `Resumable() = true`. When starting with `--start <ledger>`:

1. Galexie calls `findResumeLedger(ctx, dataStore, schema, startLedger, endLedger)`.
2. This scans the remote datastore to find the first _absent_ ledger at or after `startLedger`.
3. If the datastore already has all data up to `endLedger`, Galexie logs "data already exported" and exits cleanly (not an error).
4. If a gap is found, Galexie sets `config.StartLedger = absentLedger` and begins export from there.

### Alignment check

Before resuming, Galexie verifies the resumed ledger aligns to the `LedgersPerFile` boundary:

```go
if absentLedger > 2 && absentLedger != schema.GetSequenceNumberStartBoundary(absentLedger) {
    return NewInvalidDataStoreError(absentLedger, schema.LedgersPerFile)
}
```

If misaligned (which can happen if a previous run wrote partial data), Galexie refuses with an error suggesting to use `scan-and-fill` instead.

### Range boundary alignment

Before export, `adjustLedgerRange()` snaps `StartLedger` and `EndLedger` to file boundaries:

```go
config.StartLedger = schema.GetSequenceNumberStartBoundary(config.StartLedger)
config.StartLedger = max(2, config.StartLedger)
if config.EndLedger != 0 {
    config.EndLedger = schema.GetSequenceNumberEndBoundary(config.EndLedger)
}
```

### No Stellar checkpoint dependency

Galexie uses Captive Core (via `ledgerbackend.NewCaptive`) for live ledger streaming. Captive Core respects checkpoint boundaries internally. Galexie itself does not checkpoint-scan the history archives for its resume logic - it queries the _datastore_ (S3/GCS) directly to find missing files.

## 5. CLI Commands and Backfill Flags

### Available subcommands

| Command         | Mode         | Bounded Required                | Use Case                              |
| --------------- | ------------ | ------------------------------- | ------------------------------------- |
| `append`        | `Append`     | No (unbounded = live streaming) | Backfill + live ingestion             |
| `scan-and-fill` | `ScanFill`   | Yes                             | Fill gaps in existing data lake       |
| `replace`       | `Replace`    | Yes                             | Re-export, overwriting existing files |
| `detect-gaps`   | `DetectGaps` | Yes                             | Audit: find missing ledger ranges     |

### Common flags (all subcommands)

| Flag            | Short | Type   | Description                                        |
| --------------- | ----- | ------ | -------------------------------------------------- |
| `--start`       |       | uint32 | Starting ledger (inclusive), must be > 1           |
| `--end`         |       | uint32 | Ending ledger (inclusive); omit or `0` = unbounded |
| `--config-file` |       | string | Path to TOML config (default: `config.toml`)       |

### Backfill example (single bounded range)

```bash
galexie append \
  --start 2 \
  --end 50457424 \
  --config-file pubnet-config.toml
```

### Parallelism for backfill

Galexie has **no built-in parallelism flag**. Parallelism is achieved by running multiple independent processes, each covering a non-overlapping ledger range:

```bash
# Worker 1: ledgers 2 - 10,000,000
galexie append --start 2 --end 10000000 --config-file config.toml &

# Worker 2: ledgers 10,000,001 - 20,000,000
galexie append --start 10000001 --end 20000000 --config-file config.toml &
```

The SDF ran 40+ parallel instances to backfill 10 years of Pubnet history in under 5 days at ~$600 compute cost. Single instance would take ~150 days.

### History archive URL configuration

No CLI flag; set in `config.toml` under `[stellar_core_config]`:

```toml
[stellar_core_config]
  network = "pubnet"
  # OR override explicitly:
  history_archive_urls = ["https://history.stellar.org/prd/core-live/core_live_001"]
```

Using `network = "pubnet"` auto-populates SDF's public history archive URLs from the Go SDK constants.

### `scan-and-fill` for gap repair

```bash
galexie scan-and-fill \
  --start 2 \
  --end 50457424 \
  --config-file pubnet-config.toml
```

Scans the entire range, exports only missing ledgers.

> **Note**: The source code defines `defaultNumWorkers = 32` in `internal/app.go`, but this constant is used exclusively by the `detect-gaps` scanner, not by `scan-and-fill`. The internal parallelism of `scan-and-fill` is not documented in public sources.

### `detect-gaps` for auditing

```bash
galexie detect-gaps \
  --start 2 \
  --end 50457424 \
  --output-file gaps-report.json \
  --config-file pubnet-config.toml
```

Outputs JSON (format matches official Stellar docs detect-gaps example):

```json
{
  "scan_from": 2,
  "scan_to": 50457424,
  "duration_seconds": "272.1s",
  "report": {
    "min_sequence_found": 2,
    "max_sequence_found": 50457424,
    "total_ledgers_found": 50457400,
    "total_ledgers_missing": 23,
    "gaps": [...]
  }
}
```

## 6. Soroban Mainnet Activation Ledger

**Protocol 20 (Soroban) activated on Stellar Mainnet on February 20, 2024.**

The first ledger running Protocol 20 is **ledger 50457424**.

- The architecture docs reference ~50,692,993 - this appears to be an approximate figure from a later point (possibly when the phased rollout reached full capacity, or from when the docs were written).
- The canonical activation ledger for filtering "Soroban-enabled" ledgers is **50457424**.
- To be safe, any backfill scan should start from ledger **2** if full history is needed, or from **50457424** for Soroban-only data.

Sources:

- [dev.to/row-bear: "RPC for Soroban mainnet"](https://dev.to/row-bear/rpc-for-soroban-mainnet-36eo) — states: "The validators cast their votes at 17:00 UTC, and the first ledger after that (50457424) was running on protocol 20."
- [stellar.org: "Protocol 20 and smart contracts are live on mainnet"](https://stellar.org/blog/developers/protocol-20-and-smart-contracts-are-live-on-mainnet) — confirms Feb 20, 2024 activation date (does not cite the exact ledger number)
- [Google Groups: stellar-validators "Stellar Core v20.2.0 / Feb. 20 Mainnet upgrade vote"](https://groups.google.com/g/stellar-validators/c/nKOhJfY29L4) — validator coordination thread for the Feb 20 vote

> **Verification caveat**: The exact ledger number 50,457,424 comes from a community blog post, not from official SDF documentation. The official Stellar blog confirms the date (Feb 20, 2024) but not the ledger number. The Horizon public instance does not retain history back to this ledger. For production use, this number should be cross-verified against a full-archive Horizon instance or stellarchain.io explorer before being hardcoded.

## 7. Backfill + Live Ingestion Coordination (Duplicate Avoidance)

Galexie is designed for **idempotent, file-based output**. There is no central coordinator; coordination is implicit through S3 key naming:

### Deduplication strategy

1. **`append` mode**: Each process scans the datastore for missing files at startup and only writes what's absent. Two processes covering overlapping ranges will each call `findResumeLedger` and skip files already present.

2. **`replace` mode**: Explicitly overwrites. Not used for concurrent backfill.

3. **`scan-and-fill` mode**: Checks existence of each file before writing - only uploads missing ones.

4. **File atomicity**: Each XDR file is a complete, self-contained unit (one or more ledgers). Uploads are atomic from S3's perspective. A partial failure leaves a partial file absent (not corrupted), which will be retried.

### Recommended pattern for concurrent backfill + live

```
Instance A: append --start 2 --end 50457424        # historical backfill
Instance B: append --start 50457424                 # live streaming (unbounded)
```

- Instance B will immediately start writing from the latest checkpoint ledger and stream forward.
- Instance A fills historical gaps working toward Instance B's position.
- When Instance A reaches the range Instance B has already covered, `findResumeLedger` returns 0 (all data present) and Instance A exits cleanly with "data already exported."
- No explicit coordination, locking, or deduplication logic needed between instances.

### Deduplication at consumer level

Since files are immutable XDR bundles stored at deterministic keys, consumers reading from S3 can rely on the key structure for deduplication. The same ledger sequence always maps to exactly the same S3 key (given fixed schema config), so duplicate reads are impossible if the schema doesn't change.

## 8. TOML Config Reference (S3, Pubnet, 1 ledger/file)

```toml
admin_port = 6061

[datastore_config]
  type = "S3"

[datastore_config.params]
  destination_bucket_path = "my-bucket/ledgers/pubnet"
  region = "us-east-1"

[datastore_config.schema]
  ledgers_per_file = 1
  files_per_partition = 64000

[stellar_core_config]
  network = "pubnet"
```

> **Note**: `CAPTIVE_CORE_STORAGE_PATH` (on-disk path for bucket files) is set via **environment variable**, not in this TOML. The `[buffered_storage_backend_config]` section does NOT exist in Galexie — it belongs to Stellar RPC's configuration. Do not include it in Galexie config.

## Key Findings Summary

| Question                     | Answer                                                                                                                       |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| XDR content type             | `LedgerCloseMetaBatch` — container holding 1+ `LedgerCloseMeta` entries (count = `ledgers_per_file`)                         |
| S3 key format                | `{hex(FFFFFFFF - partition_start)}--{part_start}-{part_end}/{hex(FFFFFFFF - file_start)}--{file_start}[-{file_end}].xdr.zst` |
| Compression                  | zstd only (`.xdr.zst`), hardcoded since v23.0                                                                                |
| Ledgers per file             | Configurable (default recommended: 1; batch option: 64)                                                                      |
| Checkpoint restart           | `append` mode scans S3 for first missing ledger and resumes there; alignment to `LedgersPerFile` boundary enforced           |
| Backfill CLI                 | `galexie append --start 2 --end <N> --config-file config.toml`                                                               |
| Parallelism                  | No built-in flag; run multiple processes on non-overlapping ranges                                                           |
| History archive URL          | Set via `network = "pubnet"` or explicit `history_archive_urls` in TOML                                                      |
| Soroban activation ledger    | **50457424** (Protocol 20, Feb 20, 2024)                                                                                     |
| Backfill + live coordination | Implicit via idempotent file writes; each process skips already-present files; no locking needed                             |
